# Скиллы Landing Image Pipeline Agent

Сводная таблица по четырём image-скиллам, которые оркестрирует
[`landing-image-pipeline-agent`](../.claude/agents/landing-image-pipeline-agent.md).
Порядок — по ходу пайплайна:

```
audit (какие слоты) → source (взять картинки) → storage (залить, локатор) → placeholder (вписать в SITE_DATA)
```

| Скилл | Краткое описание | Зона ответственности |
|-------|------------------|----------------------|
| **`landing-image-audit`** | Сканер слотов-изображений в `SITE_DATA` (read-only) | **Что нужно и куда.** Рекурсивно обходит `window.SITE_DATA`, находит каждый слот со значением-картинкой и выдаёт таблицу `mock (slot) → ссылка` с пустой правой колонкой. Только диагностика — `data.js` не правит, картинки не достаёт. Даёт скелет рабочего списка. |
| **`landing-image-source`** | Добытчик картинок из стоковых источников | **Откуда взять.** По поисковому `query` ищет кандидатов (`search`) и скачивает выбранное в `pages/<landing>/img/` (`fetch`). Владеет **source-list** (откуда: `unsplash`…). Кладёт картинку на диск — в хранилище не заливает, к слотам не привязывает. |
| **`landing-image-storage`** | Менеджер изображений через сменные хранилища | **Куда положить и какая ссылка.** Заливает картинку (`upload`), скачивает обратно (`download`), отдаёт локатор/список (`locate`). Владеет **storage-list** (куда: `vercel` через Blob / `local` через ФС). Возвращает локатор (URL для vercel, путь для local). К `SITE_DATA` не привязывает. |
| **`landing-image-placeholder`** | Писарь ссылок в слоты `SITE_DATA` (пишущая пара к audit) | **Вписать ссылку в слот.** Берёт ту же таблицу `mock → ссылка` (уже заполненную) и записывает каждый URL в нужный слот `pages/<landing>/js/data.js`. Пишет только в существующие img-слоты, формат `data.js` сохраняет. Сам картинки не достаёт и не заливает — только привязывает готовые ссылки. |

## Логика связки

`audit` говорит *что нужно* → `source` *берёт* → `storage` *хранит и даёт ссылку* →
`placeholder` *вписывает*. Агент сам деталей источника/хранилища не исполняет —
**делегирует** их скиллам и ведёт между ними рабочий список `mock → ссылка`.

## Две оси перпендикулярности

- **source ↔ storage** — «откуда взять» против «куда положить» (разные вопросы,
  разные list'ы: source-list vs storage-list).
- **audit ↔ placeholder** — один **читает** слоты, другой **пишет** в них
  (симметричная пара по той же таблице `mock → ссылка`).
