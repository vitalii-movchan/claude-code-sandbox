# Onboarding — коллекция лендингов

Проект — набор одностраничных маркетинговых лендингов: статика на чистых
HTML + CSS + vanilla JS (без сборки и фреймворков), каждый воспроизводит свой
макет-референс. Источник истины по проекту — `CLAUDE.md` в этой папке; здесь —
короткая карта «с чего начать».

## Раскладка

- `pages/<имя>/` — сами лендинги (`index.html`, `css/`, `js/`), самодостаточны и
  изолированы друг от друга. `pages/index.html` — витрина со ссылками на все.
- `ui/<имя>/` — макеты-референсы под каждый лендинг.
- `.claude/skills/`, `.claude/agents/` — доменные скиллы и их агенты-обёртки.

Каждый лендинг построен по **data-driven** паттерну: `index.html` — каркас с
пустыми контейнерами списков, `js/data.js` — контент (`window.SITE_DATA`),
`js/main.js` — рендерит данные в контейнеры. Детали — в `CLAUDE.md`.

## Запуск

Только через локальный HTTP-сервер из `pages/` (не из `file://`):

```bash
cd pages && python3 -m http.server 8000
# лендинг: http://localhost:8000/<имя>/
```

## Скиллы — с чего начать по задаче

Жизненный цикл лендинга разнесён на три роли — **создаёт → диагностирует →
чинит**; вызываются как `/<имя>` или авто-подбором по описанию задачи:

| Задача | Скилл | Аргумент |
|---|---|---|
| Создать лендинг по картинке-референсу | `landing-page-builder` | — (медиа в запросе) |
| Проверить готовый лендинг, найти дефекты (без правок) | `landing-page-audit` | имя папки в `pages/` |
| Почистить/отрефакторить готовый лендинг | `landing-page-refactor` | имя папки в `pages/` |
| Открыть лендинг в браузере, снять скриншот | `landing-page-preview` | имя папки в `pages/` |

Роли разведены: **builder** строит с нуля, **audit** только диагностирует и
отчитывается (ничего не меняя), **refactor** чинит найденное по каталогу audit.
Источники истины единственны: каталог дефектов — у `audit`
(`references/defect-catalog.md`), архитектурный образец — у `builder`. У `builder`
и `refactor` есть агенты-обёртки (`*-agent`) для запуска отдельной задачей; у
`audit` обёртки нет — он самостоятелен (ревью, гейт перед деплоем) и обычно
вызывается как шаг refactor либо вручную. `landing-page-preview` — общая механика
запуска в браузере (сервер → Chrome → скриншот → консоль → стоп); на неё опираются
все три роли для верификации / живой картины, обёртки у неё нет. Плюс
`copywriting` — для текстов лендингов.

Типовой поток: **builder** создаёт лендинг и через **preview** сверяет результат
с референсом; позже **audit** находит дефекты, **refactor** чинит их и через
**preview** проверяет, что вид не изменился.

## Изображения — отдельная подсистема

Перпендикулярно жизненному циклу лендинга есть набор скиллов про **наполнение
картинками** слотов `SITE_DATA` — каждый владеет своим узким контрактом:

| Роль | Скилл | Контракт |
|---|---|---|
| Какие слоты нужны (read-only) | `landing-image-audit` | `SITE_DATA` → таблица `mock → ссылка` |
| Откуда взять картинку | `landing-image-source` | `search` / `fetch` (стоки: unsplash…) |
| Куда положить, какая ссылка | `landing-image-storage` | `upload` / `download` / `locate` (vercel/local) |
| Вписать ссылку в слот | `landing-image-placeholder` | таблица → `data.js` |

Сквозной проход `audit → source → storage → placeholder` оркестрирует агент
**`landing-image-pipeline-agent`** (принимает `<landing>`, опц. `source`/`storage`).
Карта-сводка — `docs/image-pipeline-skills.md`.

Подсистема прошла **аудит** (границы + корректность скриптов): каждый скилл
определён позитивно через свой контракт, не через отрицание соседей; найденные
баги (хардкод расширения, дефолт download, http-false-positive, обрезка URL при
парсинге) исправлены и проверены тестами.

## Деплой

Лендинги создаются локально, деплоятся в GitHub Actions на Vercel: push в `main`
→ production, каждый PR → preview-деплой со sticky-комментарием. Конфиги
(`deploy-vercel.yml`, `vercel.json`) живут в корне git-репо. Подробности и
секреты — в `CLAUDE.md`.

Прод (публичный): <https://project-uz58t.vercel.app> — витрина со ссылками на
все лендинги. Preview-ссылки из PR отдают 401 (Vercel Deployment Protection,
by design).
