# Аудит лендинга

Диагностирует существующий лендинг и выдаёт **отчёт о дефектах** — что не так,
где, почему и как чинить кратко. **Ничего не меняет.** Это диагностика, отделённая
от лечения: находки осмыслены сами по себе (ревью, оценка перед деплоем, решение
«рефакторить или нет»), а правит код уже `landing-page-refactor` по этому отчёту.

Три роли проекта разведены чётко: **`landing-page-builder` строит** лендинг с нуля,
**`landing-page-audit` (этот) диагностирует** готовый, **`landing-page-refactor`
чинит** найденное. Audit — вход в refactor, но самостоятелен: отчёт ценен и без
последующей правки.

## Цель и границы

- **Целевой лендинг — аргумент:** имя папки в `pages/` (`car_service`,
  `move_service`…). Аудируй `pages/<аргумент>/`. **Аргумент не задан** → не
  угадывай: покажи `ls pages/` и спроси, какой именно (лендинги изолированы,
  обычно имеется в виду один).
- **Read-only — это суть роли.** Не правь ни строки: ни код, ни контент, ни
  «мелочь заодно». Ценность audit именно в том, что он даёт картину, не трогая
  лендинг — поэтому им можно проверить чужой лендинг, прогнать перед деплоем,
  решить «стоит ли рефакторить». Захотелось починить → это уже `landing-page-refactor`,
  скажи об этом в отчёте, но не делай сам.
- **Изоляция.** Каждый лендинг самодостаточен (`index.html`, `css/styles.css`,
  `js/data.js`, `js/main.js`). Аудируй строго целевую папку; повтор технического
  data-driven паттерна между лендингами — норма по дизайну, не дефект.

## Процедура

1. **Прочитай все четыре файла целиком** — диагностируй по фактическому коду, не
   по чужим номерам строк. Крупный лендинг / несколько сразу → раздай сабагентам
   (по одному на лендинг, read-only), сведи находки и **перепроверь** их:
   сабагенты выдумывают строки.
2. **Пройди по четырём осям** — полный каталог дефектов с признаками
   распознавания (grep-сигнатуры, на что смотреть, почему это проблема) в
   `references/defect-catalog.md`. Это профильное знание скилла — читай его как
   репертуар проверок.
3. **Быстрые grep-сигнатуры** для старта (детали и остальные — в каталоге):
   ```bash
   grep -rn '{[[:space:]]*}' pages/<имя>/css/*.css   # пустые селекторы
   grep -rn 'style=' pages/<имя>/index.html          # инлайн-стили
   node --check pages/<имя>/js/main.js                # синтаксис (битый = пустая страница)
   ```
4. **(Опционально) живая картина** — если нужно увидеть лендинг и консоль, запусти
   через скилл **`landing-page-preview`** (синтаксис JS → сервер из `pages/` →
   Chrome → скриншот → консоль → стоп). Для чистой диагностики кода не обязательно,
   но консоль ловит рантайм-дефекты (молчаливые 404, ошибки JS), которых в коде не
   видно. Ожидаемо и безвредно: `404` на демо-картинки `img/*.jpg` с `console.warn`
   от `onerror`.

## Формат отчёта

Группируй находки **по четырём осям** — так отчёт читается параллельно процедуре
и его напрямую потребляет `landing-page-refactor`. На каждую находку: **где**
(`файл:строка`), **симптом**, **почему** проблема, **как чинить** (кратко — одно
предложение, не код). Ось без находок — пометь «чисто».

```
# Аудит <имя-лендинга>

## Дубли
- `js/main.js:62,74` — два почти одинаковых рендерера (services/reviews).
  Почему: правка разметки требует синхронных изменений в двух местах.
  Фикс: вынести в один хелпер с параметрами.

## CSS
- `css/styles.css:210` — пустой селектор `.hero-old {}`.
  Почему: мёртвый код, шум при чтении. Фикс: удалить (подтвердив grep'ом).
- (чисто по остальным пунктам)

## JS
- `js/main.js:28` — загрузка фона через `new Image()` без `onerror`.
  Почему: 404 падает молча, отладка вслепую. Фикс: добавить `probe.onerror` с warn.

## HTML / a11y
- `index.html:15` — декоративный эмодзи без `aria-hidden`.
  Почему: скринридер озвучит мусор. Фикс: `aria-hidden="true"`.

## Итог
N находок (X багов, Y чистота). Чинит — `landing-page-refactor <имя>`.
```

Заверши **итогом**: сколько находок, разведи реальные баги (утечки, молчаливые
404, битый JS) и косметику (мёртвый CSS, группировка токенов) — это помогает
решить, рефакторить ли. Если лендинг чист — так и скажи, это валидный результат
аудита. **Ничего не коммить и не править** — отчёт и есть результат.
