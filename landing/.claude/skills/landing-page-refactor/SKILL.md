# Рефакторинг лендинга

Скилл приводит в порядок уже существующий лендинг, **не меняя его дизайн и
поведение**: убирает мёртвый код и дубли, чинит реальные баги, выправляет
семантику и доступность, по запросу — правит битый контент. Критерий успеха —
код стал чище и корректнее, а внешне и функционально лендинг остался тем же.

## Какой лендинг чистить — аргумент при вызове

Целевой лендинг приходит **аргументом**: имя папки внутри `pages/` (например
`car_service`, `move_service`). Это и есть «референс» — конкретный лендинг,
который скилл знает, где взять: `pages/<имя>/`.

- **Аргумент задан** → работай с `pages/<аргумент>/`.
- **Аргумент не задан** → не угадывай. Покажи список папок в `pages/`
  (`ls pages/`) и спроси, какой именно лендинг рефакторить. Чистить наугад или
  все сразу без спроса — плохо: это разные изолированные лендинги, и пользователь
  обычно имеет в виду один.

Каждый лендинг **изолирован** (см. CLAUDE.md проекта): свой `index.html`,
`css/styles.css`, `js/data.js`, `js/main.js`. Чисти строго внутри целевой
папки. **Ничего не выноси в общий слой между лендингами** — повторение
технического data-driven паттерна между разными лендингами это норма по дизайну,
а не дублирование, которое надо устранять. Цель — навести порядок ВНУТРИ одного
лендинга.

## Что входит в рефакторинг

Полный цикл: **аудит по четырём осям → чистка кода → (по запросу) починка
контента → верификация в браузере**. По умолчанию контент НЕ трогаем — только
код; тексты правим, лишь когда пользователь явно просит «и контент тоже».

### 1. Аудит по четырём осям

Прочитай все четыре файла целиком и оцени по осям. Для крупного лендинга или
когда чистишь сразу несколько — разумно раздать аудит сабагентам (по агенту на
лендинг, read-only), а находки свести. **Перепроверяй находки сабагентов по
факту**: они склонны выдумывать номера строк и предлагать фиксы, противоречащие
архитектуре (например «вынести общую функцию между лендингами» — это нарушает
изоляцию). Отбраковывай такое.

- **Дублирование внутри файла.** Повторяющиеся блоки в `main.js` (почти
  одинаковые рендереры списков → обобщить в один хелпер с параметрами); дубли
  правил в CSS (несколько селекторов с одинаковым телом → общий класс +
  различия). Между лендингами — НЕ трогать.
- **CSS-чистка.** Пустые селекторы (`.foo {}`), мёртвые правила (селектор не
  встречается в HTML/JS — проверь grep'ом перед удалением), конфликтующие
  объявления (`font-style: normal` затем `italic`), избыточные no-op (`gap: 0`
  на дефолтном flex), неупорядоченные `:root`-токены (сгруппируй: Colors /
  Typography / Layout / Shadows), объявленные, но неиспользуемые переменные.
- **JS-качество.** Переусложнённая логика рендера, повторы, утечки слушателей
  (listener вешается повторно в resize вместо одного делегирования), отсутствие
  `onerror` при загрузке фоновых картинок (молчаливый сбой при 404), границы у
  слайдеров (offset без clamp укручивается в бесконечность), безопасная сборка
  значений (`url("...")` с двойными кавычками).
- **HTML-семантика и доступность.** Здесь две разные по риску группы, и
  обращаться с ними надо по-разному (см. ниже «Чистка кода»):
  - *Безопасно — добавляй молча:* `aria-hidden="true"` на декоративные
    эмодзи/иконки; `role="img"`/`aria-label` на осмысленные иконки и
    плейсхолдеры-картинки; `<input>` без label/aria-label; инлайн-стили в HTML,
    которым место в CSS. Это добавление атрибутов/перенос стиля — вид и раскладка
    не меняются.
  - *Рискованно — предлагай, не делай без «да»:* смена структурного тега
    (`<div>` → `<ul>/<li>`, `<figure>`, `<section>`), которая затронет CSS-раскладку.

### 2. Чистка кода

Применяй правки **по фактическому коду** (перечитай файлы, не доверяй номерам
строк из чужого аудита вслепую). Раздавай по агенту на лендинг, если чистишь
несколько — лендинги изолированы, параллельная запись безопасна.

Каждая правка обязана быть **поведенчески-нейтральной**: вид и работа лендинга
не меняются. Перед удалением «мёртвого» — подтверди grep'ом, что оно правда не
используется. Если вводишь общий класс/хелпер — согласуй имена между
HTML/CSS/JS, чтобы ничего не рассыпалось.

**Граница «чиню молча» vs «предлагаю».** Рефакторинг здесь — не редизайн.
Бесспорное и обратимое (мёртвый код, дубли, утечки слушателей, `onerror`,
группировка `:root`, конфликты CSS-правил, добавление a11y-атрибутов) — чини
сразу. А смену структурного тега, которая затронет CSS-раскладку (`<div>` →
`<ul>/<li>` и т.п.), **не делай молча — вынеси в отчёт отдельным пунктом и
спроси**. Семантичный `<ul>` лучше `<div>`, но это *улучшение*, а не
*исправление*: тег тянет браузерные дефолты (`list-style`, отступы), а сетка
часто завязана на текущую структуру — риск тихо сдвинуть вёрстку есть, а выгода
вкусовая. Формулировка-образец: «`services-list` семантически — список; перевод
в `<ul>/<li>` поднимет доступность, но затронет CSS-раскладку. Сделать? По
умолчанию рабочую вёрстку не трогаю.» Это та же логика, что и с контентом ниже:
нейтральное — молча, рискованное — со спросом. (Свежесгенерированные скиллом
`landing-page` лендинги уже семантичны — так что обычно тут чинить нечего.)

### 3. Починка контента (только по запросу)

Тексты в `index.html` и `js/data.js` по умолчанию **не трогай** — это отдельная
ответственность. Чини их, только когда пользователь явно попросил. Типичные
дефекты: опечатки в видимых заголовках, продублированные элементы массива в
`data.js`, битый/бессмысленный текст, сломанная структура данных (перепутаны
`value`/`label`, пустые обязательные поля). Где правка контента не однозначна
(чем заменить, удалить дубли или дописать новые) — **спроси**, не сочиняй
отсебятину.

### 4. Верификация в браузере (обязательный финал)

Чистка не должна ничего сломать — это надо увидеть, а не предположить.

1. Синтаксис JS:
   ```bash
   node --check pages/<имя>/js/main.js && node --check pages/<имя>/js/data.js
   ```
2. Проверь, что не осталось мусора, который чистил (примеры):
   ```bash
   grep -rn '{[[:space:]]*}' pages/<имя>/css/*.css   # пустые селекторы
   grep -rn 'style=' pages/<имя>/index.html          # инлайн-стили
   ```
3. Подними сервер из `pages/` и открой лендинг через chrome-devtools MCP:
   ```bash
   cd "<корень-репозитория>/pages" && python3 -m http.server 8000
   # http://localhost:8000/<имя>/
   ```
   - `navigate_page` на лендинг, `list_console_messages` — новых ошибок быть не
     должно. **Важный нюанс:** `console.warn` от добавленного `onerror` на
     отсутствующих демо-картинках (`img/*.jpg` 404) — это ОЖИДАЕМОЕ поведение,
     а не регрессия: код корректно логирует и оставляет CSS-плейсхолдер.
   - `take_screenshot` затронутых рефакторингом секций (и тех, где правил
     контент) — убедись, что они выглядят как прежде / корректно.
   - Останови сервер после проверки (`TaskStop`).

## Отчёт

Заверши кратким отчётом по осям: что изменил (с привязкой к файлам), что
проверил и **сознательно НЕ менял с причиной** (например «`--color-dark-2`
оставлен — используется в 3 местах»; «контент не трогал — не просили»). Явный
список несделанного так же важен, как список сделанного: он показывает, что
находки рассмотрены, а не пропущены.

Изменения не коммить, если пользователь не попросил.
