# Рефакторинг лендинга  

Приводит существующий лендинг в порядок, **не меняя дизайн и поведение**:
мёртвый код и дубли — вон, реальные баги — чинить, семантика и доступность —
выправить, контент — по запросу. Успех = код чище и корректнее, а внешне и
функционально лендинг тот же.

## Цель и границы

- **Целевой лендинг — аргумент:** имя папки в `pages/` (`car_service`,
  `move_service`…). Работай с `pages/<аргумент>/`. **Аргумент не задан** → не
  угадывай: покажи `ls pages/` и спроси, какой именно (это разные изолированные
  лендинги, обычно имеется в виду один).
- **Изоляция.** Каждый лендинг самодостаточен (`index.html`, `css/styles.css`,
  `js/data.js`, `js/main.js`). Чисти строго внутри целевой папки и **ничего не
  выноси в общий слой между лендингами** — повтор технического data-driven
  паттерна между ними это норма по дизайну, а не дублирование.

Полный цикл: **аудит по 4 осям → чистка кода → (по запросу) контент →
верификация в браузере.**

## 1. Аудит по четырём осям

Прочитай все четыре файла **целиком** — правь по фактическому коду, не по
номерам строк из чужого аудита. Если чистишь несколько лендингов / лендинг
крупный — раздай аудит сабагентам (по одному на лендинг, read-only) и сведи
находки; но **перепроверяй их**: сабагенты выдумывают строки и предлагают фиксы
против изоляции.

- **Дублирование внутри файла.** Похожие рендереры в `main.js` → один хелпер
  с параметрами; селекторы с одинаковым телом в CSS → общий класс + различия.
- **CSS.** Пустые селекторы (`.foo {}`), мёртвые правила (нет в HTML/JS),
  конфликты (`font-style: normal` затем `italic`), no-op (`gap: 0` на flex),
  неиспользуемые `:root`-токены; `:root` сгруппируй (Colors / Typography /
  Layout / Shadows).
- **JS.** Переусложнённый рендер и повторы; утечки слушателей (listener в
  resize вместо одного делегирования); нет `onerror` у загрузки фонов
  (молчаливый 404); слайдер без clamp границ; `url("...")` с двойными кавычками.
- **HTML / a11y.** Декоративным эмодзи/иконкам — `aria-hidden`; осмысленным
  иконкам и плейсхолдерам-картинкам — `role="img"`/`aria-label`; `<input>` без
  label; инлайн-стили → в CSS. Несемантичные `<div>` вместо `<ul>/<li>` и т.п. —
  кандидат на правку, но с оговоркой из §2.

## 2. Чистка кода

Каждая правка **поведенчески-нейтральна**: вид и работа не меняются. Перед
удалением «мёртвого» — подтверди grep'ом. Вводишь общий класс/хелпер — согласуй
имена в HTML/CSS/JS.

**Граница «молча vs со спросом»** (рефакторинг ≠ редизайн):

- *Молча:* мёртвый код, дубли, утечки, `onerror`, группировка `:root`, конфликты
  CSS, добавление a11y-атрибутов — бесспорно и обратимо.
- *Со спросом:* смена структурного тега, затрагивающая CSS-раскладку (`<div>` →
  `<ul>/<li>`, `<figure>`…). Это *улучшение*, а не *исправление*: тег тянет
  браузерные дефолты, а сетка часто завязана на структуру. Вынеси в отчёт
  отдельным пунктом и спроси — по умолчанию рабочую вёрстку не трогай.

(Лендинги, сгенерированные скиллом `landing-page`, уже семантичны — тут обычно
чинить нечего.)

## 3. Контент (только по запросу)

Тексты в `index.html` и `data.js` по умолчанию **не трогай**. Чини, лишь когда
просят. Дефекты: опечатки в заголовках, дубли элементов массива, битый текст,
сломанная структура данных (перепутаны `value`/`label`, пустые поля). Где правка
не однозначна (чем заменить, удалить дубли или дописать) — **спроси**.

## 4. Верификация в браузере (обязательно)

1. `node --check pages/<имя>/js/main.js && node --check pages/<имя>/js/data.js`
2. Проверь остатки: `grep -rn '{[[:space:]]*}' pages/<имя>/css/*.css` (пустые
   селекторы), `grep -rn 'style=' pages/<имя>/index.html` (инлайн-стили).
3. Сервер из `pages/` + chrome-devtools MCP:
   `cd <репо>/pages && python3 -m http.server 8000` → `http://localhost:8000/<имя>/`
   - `list_console_messages` — новых ошибок быть не должно. `console.warn` от
     `onerror` на отсутствующих демо-картинках (`img/*.jpg` 404) — **ожидаемо**,
     не регрессия.
   - `take_screenshot` затронутых секций — вид как прежде.
   - Останови сервер (`TaskStop`).

## Отчёт

Кратко по осям: что изменил (с файлами), что проверил, что **сознательно НЕ
менял и почему** (напр. «`--color-dark-2` оставлен — используется»; «контент не
трогал — не просили»). Список несделанного важен не меньше — он показывает, что
находки рассмотрены. Изменения не коммить без просьбы.
