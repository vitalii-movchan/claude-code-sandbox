# Заполнение слотов-изображений лендинга

Скилл отвечает на один вопрос: *как вписать готовые ссылки в слоты `SITE_DATA`?*
`<landing>` — аргумент (имя папки в `pages/`, см. корневой `CLAUDE.md`).

Это **пишущая пара** к read-only `landing-image-audit`. Тот рекурсивно **читает**
слоты и печатает таблицу `mock (slot) → ссылка для заполнения` с пустыми
плейсхолдерами. Этот берёт **ту же таблицу с вписанными во вторую колонку
ссылками** и **записывает** каждый URL в свой слот `pages/<landing>/js/data.js`.

```
landing-image-audit   →  таблица со слотами + «← вписать»   (ЧИТАЕТ)
                          ↓ человек/соседний скилл вписывает URL
landing-image-placeholder →  записывает URL в слоты data.js  (ПИШЕТ)
```

## Вход: заполненная таблица audit

Скилл парсит markdown-таблицу — строки `| <slot> | <url> |`:

- слева — адрес слота в JS-нотации (`reviews[0].photo`, `media.hero`,
  `services[2].img`…), ровно как печатает `landing-image-audit`;
- справа — ссылка на изображение (`https://…` или путь `img/…`).

Шапка, разделитель и **незаполненные** строки (правая ячейка пуста, прочерк или
ещё держит плейсхолдер `← вписать`) пропускаются. Таблица подаётся файлом
(`--table <file>`) или на stdin.

## Что записывается (симметрично audit)

Скилл пишет URL в слот, только если слот **существует** И его **текущее**
значение проходит тот же критерий картинки, что у `landing-image-audit`
(путь `img/…`, расширение картинки или `http(s)://…`). Иначе строка
**пропускается** и попадает в отчёт «Пропущено»:

- *слот не существует* — адреса нет в `SITE_DATA`;
- *слот не является img-слотом* — есть, но текущее значение не картинка
  (не перезаписываем заголовок/текст по ошибочному адресу);
- *значение уже совпадает* — URL уже стоит, менять нечего.

Запись точечная и безопасная для `data.js` (это JS-исходник, не JSON):
заменяется ровно строковый литерал слота, формат файла — кавычки, отступы,
комментарии, соседние слоты — сохраняется. Дубли значений (одна картинка в
нескольких слотах) разводятся по позиции вхождения, как в обходе audit.

## Граница: только привязка готовых ссылок

Скилл **не достаёт** картинки из стоков и **не заливает** их в хранилище — он
лишь **привязывает уже готовые ссылки** к слотам. Перпендикулярен соседям по оси
изображений:

- **что и куда нужно** (read-only диагностика) → `landing-image-audit`;
- **получить** картинку из стока → `landing-image-source` (`search`/`fetch`);
- **залить / дать ссылку** на картинку → `landing-image-storage`
  (`upload`/`download`/`locate`);
- **вписать ссылку в слот** → этот скилл.

Замыкает пайплайн: `image-audit` (таблица слотов) → `image-source`/`image-storage`
(взять картинку, получить URL) → впиши URL во вторую колонку таблицы →
**`image-placeholder`** (запись в `data.js`).

## Процедура

1. **Определи `<landing>`** из запроса. Не задан → спроси, какой лендинг.
2. **Подготовь таблицу** — возьми вывод `landing-image-audit` и впиши ссылки во
   вторую колонку (вручную или из `image-source`/`image-storage`). Строки без
   ссылок можно оставить — они пропустятся.
3. **Прогони dry-run** — `node scripts/fill.mjs <landing> --table <file> --dry-run`
   (или подай таблицу на stdin). Посмотри отчёт: что применилось бы, что
   пропущено и почему.
4. **Примени** — без `--dry-run`. Скрипт перезаписывает только литералы слотов.
5. **Сверь** при желании повторным `landing-image-audit` или
   `landing-page-preview` (картинки в браузере).

## Скрипт

- `scripts/fill.mjs` — Node ≥ 18, без зависимостей. Грузит `data.js` в
  изолированном `vm`-контексте (как audit-сканер), резолвит адреса слотов,
  валидирует их по тому же `IMAGE_RE` и применяет замены литералов разом по
  исходнику. Флаги: `--table <file>` (иначе stdin), `--pages <dir>` (по умолчанию
  поднимается от папки скилла к `landing/pages`), `--dry-run` (показать без
  записи). Печатает markdown-отчёт: применённые слоты + блок «Пропущено».
