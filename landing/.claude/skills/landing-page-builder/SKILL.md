# Генератор лендингов по референсу

Скилл собирает одностраничные лендинги, **буквально повторяя приложенный
референс**: компоновку, набор и порядок секций, кнопки и подписи, палитру,
типографику. У каждого лендинга свой вид — единого шаблона нет; постоянна только
техническая основа.

## Референс — источник истины

Главный вход — **изображение-референс**. Изучи его через Read и разбери
буквально: секции и их порядок; элементы каждой (заголовки, тексты, карточки,
списки, формы); подписи и назначение кнопок; палитру (фон, акцент, текст);
типографику (засечки/гротеск, вес, регистр). Воспроизведи именно этот дизайн —
не заменяй «своим красивым» и не упрощай до типового.

**Референс не приложен** → попроси картинку. Если пользователь настаивает —
выведи структуру и палитру из сути бизнеса, привлекая принципы скилла
`frontend-design`, и предупреди, что без референса точность не гарантирована.

**Текст лендинга** заполняй осмысленно: нужно знать что за бизнес, для кого и
какое целевое действие (позвонить, оставить заявку, купить, скачать). Обычно
это в запросе; если нет — задай 1–2 коротких вопроса.

## Техническая основа (одинакова для всех лендингов)

Статический сайт без сборки, пакетного менеджера и фреймворка. Структура:

```
<выходная-папка>/
├── index.html        каркас секций (по референсу, семантичный); контейнеры списков пустые
├── css/styles.css    стили под референс; палитра — токены в :root
└── js/
    ├── data.js       единственный источник контента: window.SITE_DATA
    └── main.js       рендерит SITE_DATA в контейнеры + мобильное меню
```

Правила, на которых держится связность:

- **Порядок скриптов.** Сначала `data.js`, потом `main.js`, оба с `defer`.
- **`escapeHtml()`** для любой строки из `SITE_DATA` перед вставкой в `innerHTML`.
- **`data.js` — единственный источник контента.** Любые списки (услуги, отзывы,
  команда, цены…) живут в `window.SITE_DATA` как массивы; в HTML контейнеры
  (`<… id="...-list">`) оставляй **пустыми** — их наполняет `main.js`. Не
  вписывай элементы списков руками в HTML.
- **Семантика — сразу, не на потом.** Подбирай тег по смыслу, а не `<div>` по
  умолчанию (список → `<ul>`/`<li>`, навигация → `<nav>` и т.д.); контейнер
  списка тоже семантичен. При генерации это бесплатно, при рефакторинге — риск.
  Теги, ARIA для иконок — в `references/architecture.md`.
- **Медиа — через `SITE_DATA`.** Путь к фото в `SITE_DATA` (`img: "img/<имя>.jpg"`),
  до появления файла — заглушка в тон референса; подмена на реальное фото = просто
  положить файл, без правки HTML/CSS. Детали — в `references/architecture.md`.
- **Палитра — CSS-переменные в `:root`** (минимум акцентный цвет); значения с
  референса.
- **Favicon — сразу, инлайн.** Браузер всегда рисует значок вкладки; без своего —
  дефолтная заглушка. Дай favicon из того, что уже есть: акцентный цвет под фон +
  знак из логотипа (монограмма или эмодзи), inline SVG в `data:`-URI в `<head>`.
  Образец и оба варианта знака — в `references/architecture.md`.
- **Запуск через локальный HTTP-сервер** (`http://localhost`), не из файловой
  системы. Пути относительные. Полноценный origin снимает ограничения `file://`:
  `iframe`-embed, `fetch`, ES-модули работают штатно — используй свободно по
  референсу.

Имена CSS-классов, набор секций и точки перелома скилл не навязывает — вёрстку и
адаптивность делай по-конкретному референсу.

## Куда сохранять

- **Путь указан** — создавай файлы там (`mkdir -p`, если нужно); непустую папку
  предупреди перед перезаписью.
- **Путь не указан** — `pages/<имя-бизнеса>/` в корне репозитория.

## Как генерировать

1. Определи выходную папку и изучи референс (структура, палитра, бизнес/аудитория/CTA).
2. Сверстай `index.html` по структуре референса; контейнеры списков — пустые.
   В `<head>` добавь inline-SVG favicon (акцентный цвет + знак из логотипа).
3. Опиши стили в `css/styles.css` под палитру и вид референса (палитра — токены в `:root`).
4. Заполни `js/data.js` контентом и путями к медиа.
5. Напиши рендереры в `js/main.js` (по блоку на список; `escapeHtml`; фолбэк для отсутствующих фото).

Технический образец обвязки (контейнер → массив → рендер, `escapeHtml`,
мобильное меню) — в `references/architecture.md`. Это образец **механики, а не
дизайна**: секции, классы и стиль бери с референса.

Пиши сразу без типовых дефектов (дешевле не породить, чем чинить потом):
`references/fix-patterns.md` — превентивный справочник «дефект → как не создать
его при генерации» по тем же четырём осям, что ищет аудит (дубли / CSS / JS /
HTML-a11y). Пробегись по нему перед финальной сверкой.

## Проверка и показ результата

Запуск в браузере и скриншоты — через скилл **`landing-page-preview`** (синтаксис
JS → сервер из `pages/` → открыть в Chrome → fullPage-скриншот десктоп + мобильный
→ консоль → погасить сервер). Здесь — что делать с результатом.

### Сверка с референсом (обязательный финальный шаг)

Критерий готовности — буквальное воспроизведение референса. Получив через
`landing-page-preview` скриншоты (десктоп + мобильный) и чистую консоль,
сопоставь их с референсом по чек-листу:

- [ ] Набор секций и их порядок сверху вниз.
- [ ] Палитра (фон, акцент, текст) — сверь по `:root`-токенам.
- [ ] Типографика: начертание, относительные размеры, вес.
- [ ] Кнопки/CTA: текст, форма, расположение.
- [ ] Композиция секций (колонки, выравнивание, что слева/справа).
- [ ] Повторяющиеся блоки отрендерены из `SITE_DATA`.
- [ ] Мобильная версия и меню-бургер соответствуют мобильному виду референса.
- [ ] Favicon во вкладке — не дефолтная заглушка: виден знак на акцентном фоне.

Нашёл расхождение → исправь и пересними. Не завершай работу с известными
расхождениями. Если расхождение вызвано конфликтом с техническими правилами —
приоритет у правил; сообщи пользователю о компромиссе. Останови сервер после
сверки.
