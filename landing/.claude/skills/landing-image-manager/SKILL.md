# Менеджер изображений лендинга

Профиль решает *куда/откуда*, действие — *что*. `<landing>` — аргумент всех
действий (см. корневой `CLAUDE.md`).

## Модель: action-list × profile-list

Два списка в двух папках, перпендикулярные друг другу:

- **`actions/`** — **action-list**, общий контракт (интерфейс), по файлу на действие:
  - `upload-action` — загрузить изображение в хранилище;
  - `download-action` — скачать изображение из хранилища;
  - `locate-action` — ссылка на изображение / список изображений.
- **`profiles/`** — **profile-list**, реализации контракта, по файлу на профиль:
  - `vercel-profile` — Vercel Blob (REST API);
  - `local-profile` — файловая система.

Профиль **наследует** интерфейс, но **реализует** каждое действие по-своему =
набор секций, по секции на действие. Контракт читай первым (`actions/`), чтобы
знать, чего ждать от любой реализации; обзор реализаций — `profiles/profile-list.md`.

## Диспетчеризация: (action, profile) → секция

Запрос задаёт **пару** `(action, profile)`. Скилл открывает файл профиля и
читает одноимённую секцию:

```
profiles/vercel-profile.md   →  ## upload-action / ## download-action / ## locate-action
profiles/local-profile.md    →  ## upload-action / ## download-action / ## locate-action
```

Например `(upload, vercel)` → `profiles/vercel-profile.md` § `upload-action`.
Никакого кода-роутера: пара имён прямо адресует секцию в файле профиля.

## Профиль по умолчанию

Профиль **не задан** → не угадывай молча: спроси, какой (vercel или local).
Если по контексту очевидно (упомянут Vercel/Blob/URL → `vercel`; путь в `pages/`
или «локально» → `local`) — выбери и скажи, какой взял.

## Процедура

1. **Определи `(action, profile)`** из запроса. Профиль неясен → § «Профиль по
   умолчанию».
2. **Прочитай интерфейс** — `actions/<action>-action.md` (контракт действия;
   общие правила — `actions/action-list.md`).
3. **Прочитай реализацию** — секцию `## <action>` в `profiles/<profile>.md`.
   Там команда запуска нужного скрипта, аргументы и что вернётся.
4. **Выполни** — запусти скрипт из `scripts/<profile>/`. Скрипты исполняемые,
   запускаются по месту; SKILL.md и профиль говорят когда какой и с чем.
5. **Отчитайся** результатом по контракту (URL после upload, путь после
   download, список при locate).

## Скрипты

- `scripts/vercel/{upload,download,locate}.mjs` — Node, REST API Vercel Blob,
  токен через `BLOB_READ_WRITE_TOKEN` в env (зависимостей-пакетов не требует).
- `scripts/local/{upload,download,locate}.sh` — bash поверх файловой системы.

Добавить новый профиль = новый `profiles/<name>-profile.md`, реализующий те же
три секции, плюс его скрипты в `scripts/<name>/`. Контракт в `actions/` при
этом не меняется — это и есть смысл общего интерфейса.

## Границы

Только изображения и профили: привязку к слотам (`SITE_DATA`) скилл **не делает**.
Операции адресуют один `<landing>`; соседние не трогаем.
