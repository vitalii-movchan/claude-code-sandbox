# Наполнение лендинга реальными картинками

Лендинги строятся data-driven: пути к картинкам лежат в `js/data.js` как
заглушки (`img/review-1.jpg` и т.п.), которых физически нет → `main.js`
предзагружает их и при 404 оставляет CSS-заглушку (градиент/emoji). Задача
скилла — достать реальные картинки, положить их в хранилище и прописать
получившиеся URL обратно в `data.js`. Дизайн, разметку и тексты не трогаем —
только значения медиа-слотов.

## Целевой лендинг — аргумент (он же референс)

Имя папки в `pages/`. **Не задан** → покажи `ls pages/` и спроси, какой. Лендинг
читается как референс: его `data.js` сам сообщает, какие слоты есть и что в них
ожидается — структуру не предполагаем, а обнаруживаем (она у всех разная:
`reviews[].photo`, `services[].img`, плоский `media:{hero,…}`, или слотов нет
вовсе как у `video_alliance`).

## Две оси провайдеров

Источник и хранилище — **независимы**, любая пара валидна:

- **SOURCE** — откуда взять картинку: `stock` *(default)* | `user-supplied`
- **STORAGE** — куда положить, какой URL прописать: `vercel-blob` *(default)* | `local`

Прежде чем выбирать пару — **прочитай `references/provider-profiles.md`**: там
резюме каждого провайдера (сильные/слабые стороны, предусловия, когда выбирать) и
матрица выбора. Это не формальность: у провайдеров жёсткие предусловия (ключи,
интернет), и резюме нужно, чтобы выбрать осознанно, а не наткнуться на падение.

**Выбор пары:**
1. Пользователь назвал явно (или приложил файлы → source=user-supplied) — бери это.
2. Иначе дефолт — **stock + vercel-blob**, но СНАЧАЛА проверь предусловия дефолта
   (см. ниже). Чего-то нет — не падай молча: скажи пользователю, чего не хватает,
   и предложи пару, которая сработает в его условиях (обычно user-supplied и/или
   local), ссылаясь на слабые стороны из резюме.

**Проверка предусловий перед запуском** (`.env` в корне репо `~/www/claude-code/sandbox`):
- `stock` → есть `UNSPLASH_ACCESS_KEY` или `PEXELS_API_KEY`? и есть интернет?
- `vercel-blob` → есть `BLOB_READ_WRITE_TOKEN`? установлен `@vercel/blob`? есть интернет?
- `local`, `user-supplied` → предусловий нет, работают офлайн.

Подгрузить `.env` в окружение перед вызовом скриптов:
`set -a; . ~/www/claude-code/sandbox/.env 2>/dev/null; set +a`

## Процедура

Все скрипты — в `scripts/`, у каждого единый контракт: аргументы флагами, ответ
JSON в stdout. Запускай из папки скилла. Карта модулей (что за что отвечает,
вход→выход) — таблица в `references/provider-profiles.md` (#карта-модулей).

### 1. Построй карту слотов (референс)

```
node scripts/slot_map.mjs <репо>/pages/<имя> > /tmp/<имя>-slots.json
```

Карта: для каждого слота — `jsonPath` (где в data.js), `key`, `currentValue`
(заглушка), `keywords` (для стокового поиска). **Слотов 0** → лендинг текстовый
(как `video_alliance`), наполнять нечего — так и скажи, останься.

Покажи пользователю сводку (сколько слотов, по секциям) — это контракт на работу.

### 2. Достань картинки (SOURCE)

**stock** — по одному вызову на слот, `keywords` из карты как запрос. Для слотов
вроде портретов команды (`team[].img`, label = имя человека) сток даст случайного
человека, не настоящего сотрудника — предупреди, что идентичность приблизительная
(см. слабые стороны stock). Ориентацию подбирай по слоту (hero/cover → landscape,
аватар → squarish):

```
node scripts/source_stock.mjs --query "<keywords>" --orientation landscape
# → { "file": "/tmp/...jpg", "credit": "..." }
```

**user-supplied** — попроси у пользователя папку с готовыми файлами, сопоставь со
слотами и **покажи сопоставление на подтверждение** (промахнуться на десятках
слотов легко):

```
node scripts/source_user.mjs --dir <папка> --slots /tmp/<имя>-slots.json --by name
# → { matches:[{jsonPath,file}], unmatchedSlots:[...], unusedFiles:[...] }
```

`--by name` (default) матчит по имени файла (`review-1.*` → слот с
`img/review-1.jpg`, либо `hero.*` → key=hero). Имена не бьются — `--by order`
(i-й файл к i-му слоту), но тогда сверка с пользователем обязательна.

### 3. Положи в хранилище (STORAGE) и получи URL

Для каждого добытого файла — один вызов storage-скрипта:

```
# default:
node scripts/storage_vercel_blob.mjs --file <file> --name <осмысленное-имя.jpg>
# офлайн / не в git:
node scripts/storage_local.mjs --landing <репо>/pages/<имя> --file <file> --name <имя.jpg>
# оба → { "url": "<значение для data.js>" }
```

Имя файла давай осмысленное по слоту (`review-1.jpg`, `team-michael.jpg`) — оно
видно в URL и в `img/`.

### 4. Пропиши URL в data.js

На каждый слот — точечная замена заглушки на полученный URL (data.js не
переписываем целиком, чтобы сохранить комментарии и форматирование):

```
node scripts/apply_url.mjs --landing <репо>/pages/<имя> --old "<slot.currentValue>" --new "<url>"
# → { "replaced": true }
```

`replaced:false` → заглушка уже заменена или значение не совпало; разберись, не
игнорируй. Одинаковые заглушки в нескольких слотах — иди по карте по одному слоту
за вызов (скрипт меняет первое вхождение).

### 5. Верификация в браузере (обязательно)

1. Синтаксис: `node --check pages/<имя>/js/data.js`.
2. Запусти и сними скриншоты через скилл **`landing-page-preview`** (сервер →
   Chrome → десктоп + мобильный скриншот → консоль → стоп).
3. Проверь: реальные картинки видны на месте бывших заглушек; в консоли **нет**
   404 по подменённым слотам (для local — файлы лежат в `img/`; для blob —
   CDN-URL отвечает). Если какой-то слот всё ещё 404 — он не подменился, вернись
   к шагу 4 по нему.

## Отчёт

Кратко: какую пару провайдеров выбрал и **почему** (со ссылкой на условия —
ключи/интернет/git), сколько слотов наполнил из скольких, что осталось заглушкой
и почему (напр. «team-портреты оставил заглушкой — сток даёт чужих людей, нужны
реальные фото от пользователя»), результат верификации. Для stock — упомяни, что
кадры тематические, не «точно этот объект». Изменения не коммить без просьбы.

## Гигиена git (важно — это была исходная боль)

- `storage=vercel-blob` → в `data.js` идут CDN-URL, бинарники в git не попадают. Хорошо.
- `storage=local` → картинки ложатся в `pages/<имя>/img/` и **попадут в git**. Если
  это нежелательно — предложи добавить `pages/<имя>/img/` в `.gitignore` (но
  тогда при деплое в Actions их там не будет — нужно заливать отдельно). Проговори
  компромисс с пользователем, не решай молча.
