# Резюме провайдеров

Скилл разнесён на **две независимые оси**. Любой `source` сочетается с любым `storage`:

```
source  (ОТКУДА достать картинку)        storage (КУДА положить → какой URL прописать)
  ├─ stock          (по умолчанию)         ├─ vercel-blob   (по умолчанию)
  └─ user-supplied                         └─ local
```

«Достать» и «положить» — разные обязанности. Source выдаёт **локальный файл** (скачанный
или принятый от пользователя). Storage берёт этот файл и возвращает **URL**, который скилл
пропишет в `data.js`. Поэтому stock→local, stock→blob, user→blob, user→local — все валидны.

### Список провайдеров

Все провайдеры одним перечнем (подробное резюме каждого — ниже по ссылке):

**source** (откуда взять картинку):
- [`stock`](#stock--фото-по-ключу-из-бесплатного-стока-default) *(default)* — фото по ключу из бесплатного стока (Unsplash/Pexels)
- [`user-supplied`](#user-supplied--готовые-файлы-от-пользователя) — готовые файлы от пользователя

**storage** (куда положить, какой URL прописать):
- [`vercel-blob`](#vercel-blob--заливка-в-vercel-blob-cdn-url-default) *(default)* — заливка в Vercel Blob, CDN-URL
- [`local`](#local--файлы-в-img-лендинга) — файлы в `img/` лендинга

### Описание осей

Ось — это **роль с взаимозаменяемыми реализациями** (провайдерами). Их две, и они
независимы: меняется одна — другая не затрагивается (Single Responsibility). Композиция
двух осей даёт `M + N` сущностей вместо `M × N` склеек, поэтому добавить источник или
хранилище — это +1 провайдер на нужной оси, а не переписывание комбинаций.

| Ось       | Отвечает на вопрос                       | Контракт         | Провайдеры                  | Default       |
|-----------|------------------------------------------|------------------|-----------------------------|---------------|
| `source`  | ОТКУДА взять байты картинки               | `запрос → файл`  | `stock`, `user-supplied`    | `stock`       |
| `storage` | КУДА положить файл и какой URL прописать   | `файл → URL`     | `vercel-blob`, `local`      | `vercel-blob` |

Стык осей — тонкий контракт данных: `source` отдаёт **файл на диске**, `storage` принимает
**файл** и возвращает **URL**. Пока стык соблюдён, провайдеры друг про друга не знают —
любой `source` сочетается с любым `storage`.

У каждого провайдера ниже — **резюме**: что умеет, сильные стороны, слабые стороны, когда
выбирать. Резюме нужно, чтобы выбор провайдера был осознанным, а не на удачу: если у
пользователя нет интернета — это сразу исключает stock и vercel-blob; если важно не тащить
бинарники в git — исключает local. Читай резюме перед выбором и проговаривай пользователю,
почему берёшь именно эту пару.

## Карта модулей

Скилл — это один оркестратор поверх независимых модулей в `scripts/`. У каждого
**одна ответственность** (одна причина для изменения), связаны только контрактом данных
`файл → URL` — друг про друга не знают. Поэтому любой source стыкуется с любым storage.

| Модуль                    | Ось     | Ответственность (одна причина меняться)            | Вход → выход            |
|---------------------------|---------|----------------------------------------------------|-------------------------|
| `slot_map.mjs`            | —       | обнаружить медиа-слоты, читая лендинг как референс  | папка лендинга → карта слотов (JSON) |
| `source_stock.mjs`        | source  | достать тематическое фото из стока по ключу         | ключевые слова → файл   |
| `source_user.mjs`         | source  | сопоставить готовые файлы пользователя со слотами    | папка + карта → matches |
| `storage_vercel_blob.mjs` | storage | залить файл в Vercel Blob                            | файл → CDN-URL          |
| `storage_local.mjs`       | storage | положить файл в `img/` лендинга                      | файл → относительный путь |
| `apply_url.mjs`           | —       | вписать полученный URL в `data.js` на место заглушки | URL + старое значение → правка |

Конвейер на один слот: `slot_map` → `source.*` (файл) → `storage.*` (URL) → `apply_url`.

## SOURCE-провайдеры (откуда картинка)

### stock — фото по ключу из бесплатного стока *(default)*

`scripts/source_stock.mjs` — по ключевым словам слота тянет фото из бесплатного стока
(Unsplash / Pexels API), скачивает кадр во временный файл.

- **Сильные стороны:** ноль ручной работы — слотов десятки, ключи берутся из семантики
  слота (соседние `title`/`name`/`alt`); фото живые и тематичные; бесплатно.
- **Слабые стороны:** нужен **интернет** на этапе сборки и **API-ключ** провайдера стока
  (`UNSPLASH_ACCESS_KEY` или `PEXELS_API_KEY` в `.env`); кадр подбирается по теме, а не
  «точно этот человек/объект» — для портретов реальных сотрудников не годится; у стоков
  есть рейт-лимиты.
- **Когда выбирать:** дефолт. Нужно быстро наполнить лендинг правдоподобными тематическими
  фото, точная идентичность объекта не важна (фон, иллюстративные снимки услуг, абстрактные
  hero). Есть интернет и ключ.

### user-supplied — готовые файлы от пользователя

`scripts/source_user.mjs` — принимает папку/набор готовых изображений от пользователя и
раскладывает их по слотам (по имени файла или по порядку, см. SKILL.md).

- **Сильные стороны:** **не нужен интернет и API-ключ**; пользователь контролирует точное
  содержимое (реальные фото команды, снятый продукт, фирменные кадры); предсказуемо.
- **Слабые стороны:** картинки надо иметь **заранее**; сопоставление файл↔слот требует либо
  договорённости об именах, либо ручного подтверждения — легко промахнуться при десятках
  слотов.
- **Когда выбирать:** есть готовые ассеты; нужна точная идентичность (портреты, конкретный
  продукт); нет интернета/ключа; бренд-гайд требует конкретных кадров.

## STORAGE-провайдеры (куда положить, какой URL прописать)

### vercel-blob — заливка в Vercel Blob, CDN-URL *(default)*

`scripts/storage_vercel_blob.mjs` — заливает файл в Vercel Blob storage через `@vercel/blob`,
возвращает публичный CDN-URL (`https://<...>.public.blob.vercel-storage.com/...`).

- **Сильные стороны:** **не попадает в git** — в `data.js` идёт абсолютный CDN-URL, бинарники
  не раздувают репозиторий и не едут в Actions; быстрая раздача с CDN; переживает пересборки.
- **Слабые стороны:** нужен токен `BLOB_READ_WRITE_TOKEN` (в `.env`); нужен **интернет при
  заливке**; в рантайме картинка тянется из сети — **без интернета у зрителя не загрузится**
  (но `data-bg`-fallback оставит CSS-заглушку, страница не сломается); внешняя зависимость
  от Vercel.
- **Когда выбирать:** дефолт для деплоя. Не хотим бинарники в git (это и была исходная боль),
  лендинг живёт онлайн на Vercel, есть токен.

### local — файлы в `img/` лендинга

`scripts/storage_local.mjs` — кладёт файл в `pages/<лендинг>/img/`, прописывает относительный
путь (как и ждёт текущий `data-bg`-механизм).

- **Сильные стороны:** **работает офлайн** — и при сборке, и у зрителя; ноль внешних
  зависимостей и токенов; проще некуда; relative-пути совместимы с уже существующим кодом
  лендингов один-в-один.
- **Слабые стороны:** картинки **попадают в git** (исходная боль пользователя) — если это
  нежелательно, добавь `pages/<лендинг>/img/` в `.gitignore`, но тогда при деплое в Actions
  их там не будет (нужно заливать отдельно); раздувает репозиторий.
- **Когда выбирать:** офлайн-разработка/демо; не хочется возиться с токенами; картинки лёгкие
  и их не жалко в git, либо они уже под `.gitignore` и деплой не нужен.

## Матрица выбора

| Ситуация                                             | source        | storage      |
|------------------------------------------------------|---------------|--------------|
| Дефолт: быстро наполнить, деплой на Vercel           | stock         | vercel-blob  |
| Нет интернета / нет ключей, демо локально            | user-supplied | local        |
| Есть готовые фирменные фото, деплой на Vercel         | user-supplied | vercel-blob  |
| Сток-фото, но всё держим офлайн в репо                | stock         | local        |

Если пользователь не задал явно — бери **stock + vercel-blob**, но сперва проверь
предусловия (ключи, интернет). Чего-то не хватает — скажи об этом и предложи пару, которая
сработает в его условиях, ссылаясь на слабые стороны выше.
