# CLAUDE.md

Коллекция одностраничных маркетинговых лендингов — статические сайты на чистых HTML, CSS и
vanilla JS, воспроизводящие макет-референс. Лендинги — в `pages/`, референсы — в `ui/`.

Быстрый старт для новых — `ONBOARDING.md` (карта проекта, скиллы, запуск, деплой).

## Архитектура (одинакова у всех)

Папки лендингов самодостаточны (`index.html`, `css/`, `js/`) и строятся по **data-driven** паттерну:

- `index.html` — каркас секций; контейнеры списков пустые (`<div id="...-list">`), наполняются в рантайме.
- `js/data.js` — контент: `window.SITE_DATA` с массивами под каждый список и путями к медиа.
- `js/main.js` — рендерит `SITE_DATA` в контейнеры + мобильное меню. Грузится **после** `data.js` (оба `defer`).
- `css/styles.css` — единственная таблица стилей; палитра в `:root` (минимум `--color-accent`).

Дизайн (секции, классы, типографика, палитра) у каждого лендинга свой — по его референсу.
Общими остаются только эти технические правила.

## Правила

- Один HTTP-сервер на все лендинги, относительные пути.
- Полноценный origin → `iframe`-embed, `fetch`, JSON и ES-модули работают без ограничений.

## Запуск

Через локальный HTTP-сервер, не из файловой системы:
`python3 -m http.server 8000` из папки `pages/` → `http://localhost:8000/<папка>/`.

## Деплой (Vercel через GitHub Actions)

Лендинги создаются локально, деплоятся удалённо в GitHub Actions на Vercel. Корень сайта —
`landing/pages/`, лендинги доступны как `/<папка>/` (напр. `/car_service/`); корневой
`pages/index.html` — витрина со ссылками на все лендинги.

Конфиги живут в **корне git-репо** (`~/www/claude-code/sandbox`), а не в `landing/` — Actions
работают на уровне репозитория:

- `.github/workflows/deploy-vercel.yml` — два джоба по событию: **production** (push в `main` +
  ручной `workflow_dispatch`) и **preview** (каждый `pull_request`). Деплой через Vercel CLI с
  авторизацией по токену. Preview-джоб постит/обновляет sticky-комментарий в PR с preview-URL
  (маркер `<!-- vercel-preview -->`). `concurrency` по `github.ref` — PR-деплои не отменяют production.
- `vercel.json` — статика без сборки (`buildCommand: null`), `outputDirectory: landing/pages`,
  `cleanUrls` + `trailingSlash`.

Секреты в репозитории: `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`.

Прод публичен (`https://project-uz58t.vercel.app`). Preview-ссылки отдают **401** — на проекте
включена Vercel Deployment Protection, preview виден только под логином Vercel; это by design.

## Скиллы и агенты

Сквозная конвенция: целевой лендинг скиллы принимают **аргументом** — это `<landing>`,
имя папки в `pages/` (`car_service`, `move_service`…). В описаниях ниже не повторяем.

Три роли по жизненному циклу лендинга — **создаёт → диагностирует → чинит**:

- **Создание.** Скилл **`landing-page-builder`** собирает лендинг по картинке-референсу — он источник
  истины по всему процессу (референс, медиа, сверка). Агент **`landing-page-builder-agent`** — обёртка
  над ним.
- **Диагностика.** Скилл **`landing-page-audit`** проводит аудит существующего лендинга по четырём осям
  (дубли / CSS / JS / HTML-a11y) и выдаёт отчёт о дефектах, **ничего не меняя**. Источник истины по
  «что искать» — его `references/defect-catalog.md` (каталог дефектов с признаками распознавания).
  Самостоятелен (ревью, оценка перед деплоем) и предшествует рефакторингу.
- **Рефакторинг.** Скилл **`landing-page-refactor`** ЧИНИТ найденное, без смены дизайна — опирается на
  каталог дефектов `landing-page-audit` (не дублирует его). Агент **`landing-page-refactor-agent`** —
  обёртка над ним.

Связь ролей: `audit` диагностирует → `refactor` лечит по его каталогу; `builder` строит независимо.
У `audit` агента-обёртки нет (как у `preview`) — отдельная роль, но обычно вызывается как шаг refactor
либо вручную для ревью.

Плюс общая механика — поддержка лендинга, не роль жизненного цикла:

- **Запуск в браузере.** Скилл **`landing-page-preview`** открывает лендинг из `pages/` в Chrome
  (chrome-devtools MCP) и снимает скриншоты: сервер → Chrome → fullPage десктоп + мобильный →
  консоль → стоп. На него опираются builder и refactor для шага верификации, audit — для живой
  картины (консоль ловит рантайм-дефекты); сам по себе годится для ручного «покажи лендинг в
  браузере». Агента-обёртки у него нет — это техническая процедура, не роль.
- **Изображения (хранение).** Скилл **`landing-image-manager`** грузит / качает / находит картинки
  лендинга. Две перпендикулярные оси: **action-list** (`actions/` — интерфейс из трёх действий
  `upload` / `download` / `locate`) и **profile-list** (`profiles/` — реализации хранилища:
  `vercel-profile` через Vercel Blob, `local-profile` через файловую систему). Пара `(action, profile)`
  адресует секцию в файле профиля, исполнение — скрипты `scripts/<profile>/`. К слотам `SITE_DATA` не
  привязывает. Агента-обёртки нет — техническая процедура, не роль.
- **Изображения (получение).** Скилл **`landing-image-source`** достаёт картинки из стоков — *откуда
  взять*, перпендикулярно *куда положить* у `landing-image-manager`. Те же две оси: **action-list**
  (`actions/` — `search` найти кандидатов / `fetch` скачать в `pages/<landing>/img/`) и **source-list**
  (`sources/` — реализации источников: пока `unsplash-source` через Unsplash API). Пара `(action, source)`
  адресует секцию в файле источника, исполнение — скрипты `scripts/<source>/`. Целостный пайплайн —
  `source → fetch → img/ → upload` (заливка уже у `landing-image-manager`). К `SITE_DATA` не привязывает.
  Агента-обёртки нет — техническая процедура, не роль.
